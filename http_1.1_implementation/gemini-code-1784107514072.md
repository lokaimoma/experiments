# HTTP/1.1 Implementation Specification & Verification Protocol (AI Generated)

[RFC 9110](https://www.rfc-editor.org/rfc/rfc9110.html)
[RFC 9112](https://www.rfc-editor.org/rfc/rfc9112.html)

This document establishes the strict functional requirements, boundary conditions, and architectural expectations for building a custom, low-level HTTP/1.1 server from scratch in C++. 

---

## 1. Socket Lifecycle & Resource Management
You must manage the underlying operating system network resources manually.

*   **RAII Socket Wrapper:** Raw socket descriptors (`int`) must be bound to the lifetime of a C++ class. Destructors must reliably close active descriptors.
*   **Address Resolution:** Use modern POSIX address resolution primitives (`getaddrinfo`) rather than obsolete interfaces.
*   **System Call Verification:** Every socket system call (`socket`, `setsockopt`, `bind`, `listen`, `accept`, `recv`, `send`) must have its return state verified. Failures must be handled gracefully without crashing the server process.
*   **Resource Leak Prevention:** The server must not leak socket descriptors, thread handles, or dynamically allocated memory over successive connection lifecycles.

---

## 2. Request Parsing & Validation (RFC 9112)
Your parser must process raw byte streams from the socket and reconstruct the message structure securely.

### A. The Start Line (Section 4)
*   **Format:** Exactly three tokens separated by single space characters: `Method`, `Request-Target`, and `HTTP-Version`.
*   **Line Termination:** The start line must terminate with an explicit carriage return and line feed (`\r\n`).
*   **Protocol Enforcement:** The parser must identify and reject protocols other than `HTTP/1.1`. If the version is unsupported, the server must respond with status `505 HTTP Version Not Supported`.
*   **Robustness:** Implement a configurable limit on the length of the start line to prevent buffer exhaustion attacks.

### B. Header Fields (Section 5)
*   **Structure:** Key-value pairs separated by a colon (`Name: Value`).
*   **Line Termination:** Each individual header field must terminate with `\r\n`.
*   **Whitespace Handling:** Leading and trailing whitespace around the header value must be trimmed. Whitespace between the header name and the colon is forbidden by the specification and must result in a `400 Bad Request`.
*   **Header Termination:** The entire header block is terminated by an empty line consisting solely of `\r\n`.
*   **Systemic Limits:** Establish maximum boundaries for individual header sizes and the total number of headers to prevent Denial of Service (DoS).

### C. Message Body Length (Section 6)
The server must determine the exact payload length before attempting to read the body from the socket:
*   **Content-Length:** If present, parse the value as a non-negative decimal integer. Read exactly that many bytes from the socket following the header block.
*   **No Length Indication:** If neither `Content-Length` nor `Transfer-Encoding: chunked` is present, the message body length is assumed to be zero.
*   **Conflict Resolution:** If a request contains both `Transfer-Encoding` and `Content-Length`, prioritize `Transfer-Encoding` or treat it as a malformed request (`400 Bad Request`) to prevent HTTP Request Smuggling.

---

## 3. Response Generation (RFC 9110)
All outgoing data must strictly adhere to the wire format.

*   **Status Line:** The first line of the response must follow the format: `HTTP/1.1 SP Status-Code SP Reason-Phrase CRLF`.
*   **Essential Headers:** Every response must include:
    *   `Content-Length` matching the exact byte size of the payload.
    *   `Content-Type` matching the media type of the payload.
*   **Response Body Separation:** An empty line (`\r\n`) must explicitly separate the response headers from the payload bytes.

---

## 4. Performance & Security Non-Negotiables
To ensure the implementation is robust, the code must adhere to high-performance systems programming patterns:

*   **Zero-Allocation/Minimal-Allocation Parsing:** Avoid copying or duplicating the raw request buffer during parsing. Use lightweight string reference wrappers (`std::string_view`) to point directly into the primary read buffer.
*   **Buffer Management:** Never use unsafe C-style string parsing functions (e.g., `strcpy`, `sprintf`). Use bounds-checked operations.
*   **Concurrency Foundation:** The socket loop must not block indefinitely on a single slow client. Implement a strategy (such as non-blocking sockets, event-driven loops, or basic multithreading) to handle multiple active connections.
